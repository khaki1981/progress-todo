# Architecture

## 1. ディレクトリ構成

プロジェクトは、UI・状態管理・保存処理を分離しやすい構造にします。

```text
src/
├── components/       # UIコンポーネント
├── hooks/            # 状態操作や副作用をまとめたカスタムフック
├── services/         # LocalStorageや保存処理の抽象化
├── store/            # 状態管理ロジック
├── types/            # 型定義
├── utils/            # ユーティリティ関数
├── assets/           # 画像・アイコン等
└── styles/           # グローバルスタイル
```

### 各フォルダの役割

- components
  - 画面を構成する部品を配置します。
  - 例: ProjectTabs, ProgressCircle, TodoList, TodoItem, SettingsDialog

- hooks
  - useProjects, useTodos, useStorage などのように、状態操作や副作用をまとめます。

- services
  - LocalStorage への読み書きやデータ変換を担当します。
  - 将来クラウド同期や別ストレージへ置き換えやすいように、インターフェースを意識した設計にします。

- store
  - アプリ全体の状態を保持するロジックを置きます。
  - UIから直接 LocalStorage を扱わず、この層を経由して更新します。

- types
  - Project, Todo, Settings などの共通型を定義します。

- utils
  - 日付計算、色生成、進捗計算などの補助関数を置きます。

- assets
  - 画像やアイコンなどの静的ファイルを置きます。

- styles
  - グローバルCSSや共通スタイルを置きます。

## 2. コンポーネント構成

UIは、責務ごとに小さなコンポーネントに分割します。

```text
App
├── ProjectTabs
├── ProgressCircle
├── TodoList
│   └── TodoItem
└── SettingsDialog
```

### 主なコンポーネント

- App
  - アプリ全体の土台となるコンポーネントです。

- ProjectTabs
  - プロジェクト一覧をタブ形式で表示します。

- ProgressCircle
  - 現在のプロジェクトに対する進捗率を円グラフで表示します。

- TodoList
  - Todo一覧を表示し、追加・編集・削除操作をまとめて扱います。

- TodoItem
  - 個々の Todo の表示と操作を担当します。

- SettingsDialog
  - 将来の設定画面の入り口として配置します。
  - 現在は基本構成として用意し、V1 では最小限に留めます。

## 3. データの流れ

UI から始まり、Store を経由して Service へ保存される流れを基本とします。

```text
UI
↓
Store
↓
Service
↓
LocalStorage
```

### 具体的な流れ

1. ユーザーが Todo を追加・編集・チェックすると、UI からイベントが発火します。
2. Store が状態を更新します。
3. Store から Service に保存要求が送られます。
4. Service が LocalStorage にデータを保存します。
5. アプリ再読み込み時には、Service が LocalStorage からデータを復元します。

この構造により、UI から直接 LocalStorage を扱わず、後から別保存先へ差し替えやすくします。

## 4. 状態管理

状態は、必要な情報ごとに分けて管理します。

### 保持する情報

- 現在のProject
  - 表示中のプロジェクトを管理します。

- Todo一覧
  - 現在のプロジェクト配下の Todo を保持します。

- プロジェクト一覧
  - すべてのプロジェクトの一覧を保持します。

- 設定
  - 今後の拡張を見据えて、音声設定や表示設定などを保持する想定です。
  - V1 では最小限の構成に留めます。

### 管理方針

- UI では状態を直接操作せず、Store や Hook を通じて更新します。
- 状態変更時は、保存処理も同時に実行するようにします。
- 保存処理は Service に委譲し、Store は状態管理に集中します。

## 5. LocalStorage構成

LocalStorage では、データを1つのキーにまとめる構成を基本とします。

### 保存キー

- progress-todo-data

### 保存内容

```json
{
  "projects": [],
  "activeProjectId": "",
  "settings": {}
}
```

### 設計の意図

- アプリ全体のデータを一箇所で管理しやすくします。
- 将来的にデータ構造を増やしても、キー構成を大きく変えにくくします。
- 将来クラウド同期へ移行する際にも、データの境界を分けやすくします。

## 6. 将来拡張

本構成は、将来の機能追加に対応しやすいように設計します。

### 広告

- UI 層に広告差し込み用の余地を残します。
- 画面下部にコンポーネントを挿入しやすい構造にします。
- V1 では未実装ですが、将来の拡張を見越してレイアウトの余白を確保します。

### クラウド同期

- 現在の保存処理は Service 層に閉じ込めておくため、LocalStorage から API ベースの保存へ置き換えやすいです。
- そのため、将来は Service の実装だけ差し替えればよい構造になります。

### Googleログイン

- 認証機能は、状態管理や保存処理と分離しやすい構成にしておきます。
- 将来的にログイン済みユーザー向けのデータ管理に切り替えやすいです。

### AI

- AI 機能は、UI とは切り離して追加しやすい構造にします。
- たとえば Todo の提案や自動要約などを、別モジュールとして追加できるようにします。

## まとめ

このアーキテクチャは、V1 の最小構成を実現しつつ、今後の機能追加を見据えた拡張性を重視した構成です。
